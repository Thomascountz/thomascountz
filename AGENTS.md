# Repository Guidelines

## Project Structure & Module Organization
Content lives in Jekyll collections: `_posts/` for blog entries, `_memos/` for shorter notes, `_pages/` for standalone pages, and `_drafts/` when work is not ready to publish. Layouts, includes, and custom plugins reside under `_layouts/`, `_includes/`, and `_plugins/`. Static assets such as CSS, JS, and images belong in `assets/`, while helper Ruby scripts (for memo workflows) live in `lib/`. The `_site/` directory is generated output—never edit files there directly.

## Build, Test, and Development Commands
```sh
bundle install               # install Ruby gems
bundle exec jekyll serve     # local server on http://127.0.0.1:4000 with autoreload
bundle exec jekyll build     # production build into _site/
bundle exec jekyll doctor    # sanity checks for config/content issues
```
Run `bundle exec rake post title="My Title"` or `bundle exec rake memo title="Idea"` to scaffold dated Markdown with front matter. Use `bundle exec rake copy_memo filepath=...` when importing notes with embedded images.

## Coding Style & Naming Conventions
Markdown and HTML follow standard GitHub-flavored Markdown plus Liquid tags. Favor descriptive slugs generated by the rake tasks and keep YAML front matter indented with two spaces, using ISO dates (`YYYY-MM-DD`). Ruby helpers follow the repo’s precedent of snake_case method names, single quotes, and guard clauses for parameter validation. Keep assets snake-cased and prefer hyphenated filenames for posts (`2025-08-12-topic.md`).

## Testing Guidelines
There is no standalone test suite; reliability comes from static checks and local builds. Before committing, run `bundle exec jekyll build` and fix any warnings, then `bundle exec jekyll doctor` to detect configuration drift. Preview changes via `bundle exec jekyll serve --livereload` and verify that pagination, SEO tags, and custom layouts render as expected. When adding memos with media, confirm that copied images land under `assets/images/memos/` and that paths resolve.

## Commit & Pull Request Guidelines
Recent history (`git log --oneline`) shows short, imperative messages such as `update chrome bypass post`. Keep commits focused, lowercase unless naming proper nouns, and mention affected content (`Add site footer and update About page content`). For pull requests, include: purpose summary, key screenshots for visual updates, links to related issues or posts, explicit instructions for special deploy steps (e.g., rerun `jekyll build`). Ensure automated builds (if any) pass before requesting review.
